## 3. 质量标准

### 硬性检查（必须全部通过才能 PR）

```bash
# 1. 字数检查
wc -m chapter-XX.md  # ≥ 10,000 中文字符

# 2. 结构检查（8个部分必须存在）
grep -c "痛点\|承诺\|原理\|方法\|案例\|工具箱\|误区\|7天挑战" chapter-XX.md

# 3. 禁词扫描（结果必须为空）
grep -E "视频|频道|订阅|这期|字幕|从.*提取" chapter-XX.md
```

### STYLE_GUIDE.md 软性检查清单

- [ ] 语气坚定务实（教练，非治疗师）
- [ ] 无医疗建议
- [ ] 无神化工具（"用这个就成功"）
- [ ] 无"立刻改变人生"许诺
- [ ] 案例具体可信，非夸张虚构
- [ ] 引用原则级/研究级/工具级正确分档

## 6. 风险预案

| 风险 | 概率 | 应对 |
|------|------|------|
| Claude 进程停滞 | 高 | `ps aux` 诊断 → `exec + pty:true` 重启，绝不自己写 |
| 某章素材<15个 | 中 | 降低案例粒度，转原则论证，通知用户 |
| 字数不达标 | 低 | 重跑 Claude，补充"7天挑战"和"误区"部分 |
| 禁词检查失败 | 中 | 运行替换脚本，Claude 重写问题段落 |
| 章节风格漂移 | 中 | M1 建立"风格锚定文件"，后续章节对照 |
| Git 分支冲突 | 低 | 解决冲突，重新创建 PR |

**停滞铁律**：即使停滞 18 小时，也不自己写 → 修复系统 → 重启 Claude（参考 `PM_LESSONS_LEARNED.md` 已记录的血泪教训）。

## 7. Git 工作流

### 分支策略
- 每章一个分支，格式：`chapter-XX-kebab-case-title`
- 一次只开一个分支，严格串行
- 永远不在 master 直接写作

### 提交规范
```bash
git commit -m "draft: 第X章 - [标题]（[字数]字）"    # 初稿
git commit -m "fix: 第X章 - 修复[禁词/结构/字数]"     # 修复
git commit -m "chapter: 完成第X章 - [标题]"           # 终稿
```

### PR 创建时机
- 所有检查点 C 通过后**立即创建**，不等用户询问
- 每章单独 PR，PR body 包含完整结构检查清单
